# Counter Sandbox 🧪

Это обучающий проект-песочница, демонстрирующий различные подходы к разработке Android-приложений на примере простого счетчика. Проект идеально подходит для изучения эволюции архитектурных паттернов и способов хранения данных.

## 🎯 Цель проекта
Показать, как одну и ту же задачу (увеличение/уменьшение числа) можно реализовать разными способами: от классического Android-подхода до современных паттернов.

---

## 🏗 Содержание песочницы

### 1. Архитектурные подходы
Проект разделен на несколько Activity, каждая из которых демонстрирует свой подход:

*   **`CountActivity` (Classic)**: Базовый вариант. Использует `SharedPreferences` для сохранения данных и `onSaveInstanceState` для обработки поворота экрана.
*   **`CountDelayActivity` (Problematic)**: Демонстрация проблем. Использует `Handler().postDelayed` для имитации задержки. Помогает понять, почему обновление UI напрямую из асинхронных операций внутри Activity может привести к утечкам или некорректному состоянию.
*   **`CountDelayPrActivity` (MVP + Moxy)**: Реализация на паттерне Model-View-Presenter с использованием библиотеки **Moxy**. Показывает, как Presenter берет на себя логику и сохраняет состояние View даже при пересоздании Activity.
*   **`CountDelayVmActivity` (MVVM + Jetpack)**: Современный стандарт. Использует `ViewModel`, `LiveData` и **Coroutines**. Демонстрация того, как `viewModelScope` корректно обрабатывает жизненный цикл и задержки.

### 2. Работа с базой данных и кеширование 💾
В проекте детально разобраны механизмы работы с локальными БД и стратегии обеспечения мгновенного отклика UI:

*   **`CountSqlActivity` (Low-level SQLite)**:
    *   **Как работать**: Прямое взаимодействие с `SQLiteDatabase` через `SQLiteOpenHelper`. Ручное управление транзакциями через `ContentValues`.
    *   **Кеширование**: Использует `onSaveInstanceState` для мгновенного восстановления состояния при повороте, чтобы не ждать запроса к БД. При записи применяется стратегия `CONFLICT_REPLACE`, что превращает обычную вставку в эффективное обновление существующей записи (аналог кэша в БД).

*   **`CountRoomActivity` (Modern Room ORM)**:
    *   **Как работать**: Использование декларативного подхода с `Entity`, `Dao` и `Database`. Реализован **функционал поиска по базе данных** (через `LIKE` запрос в Dao).
    *   **Многоуровневое кеширование**: 
        1.  **Первая очередь (UI Cache)**: Значение берется из `savedInstanceState` для мгновенного отображения.
        2.  **Вторая очередь (DB Source)**: Асинхронная загрузка из Room через **RxJava 3**. Если данные в БД свежее или отличаются от кеша — UI обновляется незаметно для пользователя.
        3.  **Оптимизация**: Запись в БД осуществляется с помощью фиксированного ID, что делает запись в таблицу «кеширующей». Поиск оптимизирован с помощью индексов (`indices`) в схеме таблицы.
        4.  **Умный поиск**: Реализован «живой» поиск с использованием операторов `debounce` (задержка), `distinctUntilChanged` (фильтрация повторов) и `switchMap` (отмена устаревших запросов).

*   **`SharedPreferences`**: Применяется в базовых примерах для хранения простых флагов и настроек.

### 3. Дополнительные фичи
*   **`IntroActivity`**: Экран приветствия, который вызывается автоматически через расширение `Activity.checkAndShowIntro()`.
*   **Logic Extension**: Логика показа интро реализована так, чтобы экран появлялся при каждом третьем запуске любого экрана (с использованием счетчика в Prefs).

## 🛠 Технологический стек
*   **Language**: Kotlin
*   **UI**: ViewBinding, XML Layouts
*   **Architecture**: Classic, MVP (Moxy), MVVM
*   **Async**: Coroutines, RxJava 3 (RxKotlin), Handler, LifecycleScope
*   **Database**: Room (modern), SQLite (native)
*   **Components**: ViewModel, LiveData, SavedState

## 📚 Как использовать для обучения
1.  **Сравните `CountActivity` и `CountDelayActivity`**: Посмотрите, как добавление задержки усложняет управление состоянием.
2.  **Изучите Moxy**: Обратите внимание, как `@StateStrategyType` в `CountDelayView` позволяет не терять данные при повороте.
3.  **Изучите работу с БД**: Откройте `CountSqlActivity` (ручной SQL) и сравните с `CountRoomActivity` (Room). Обратите внимание на объем кода.
4.  **Разберитесь в кешировании**: Посмотрите комментарии в `CountRoomActivity` — там описан процесс "холодного" и "горячего" старта экрана с данными.
5.  **Поиск**: Изучите, как реализован "живой поиск" в `CountRoomActivity`. Обратите внимание на использование `PublishSubject` и цепочку операторов `debounce` -> `distinctUntilChanged` -> `switchMapSingle`. Это классический паттерн для предотвращения лишних запросов к БД и решения проблем Race Condition.
